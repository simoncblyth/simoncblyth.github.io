TEX2PDFPNGÂ¶

Both the below are invoked by images-prepare-dev/images-preillus exist_putt into /db/hfagc_system are done as a side effect only when an update is required

tex2pdfpng-update

does conversions only for extras/pdgs/pdg (not qtags) doesnt change often, as infrequently need new particles/symbols

The upshot:

  1. smy tag.xml documents in file system and /db/hfagc_system/tag.xml
  2. PNG and PDF files in filesystem for each code
tex2pdfpng-qtags path

does conversions for new qtags, ie new decay modes split from the -update as these change more often and take a long time to run

the staleness comparison is done by comparing timestamps of the scripts summary file with that of the path passed in the argument which is the canonical qtag2latex.xml:

heprez-xtalk qtag2latex == /data/heprez/data/images/indices/qtag2latex.xml

tex2pdfpng-convert tag [path]

Invoked by the above two functions.

If the tag src file is newer than tag smy file at HEPREZ_DATA/images/tag/tag.xml then perl scripts from HEPREZ_HOME/images/tex2pdfpng/ are run to convert tex snippets into pdf and png and update the smy file.

The path argument is only used tag of qtags The smy file is uploaded into the eXist DB at /db/hfagc_system/tag.xml

Hmm, how does this fit with new tex2svg- based conversions

The tag can be:

qtags
src is the second path argument
extras
depends on the list of codes and corresponding latex in the extras.pl, src is the extras.pl script (so caution regards touching that)
pdgs
depends on the parsing of the initial particle latex, src is the pdg.store
pdg
depends on the parsing of the initial particle latex, src is the pdg.store

The resulting extras.xml and pdgs.xml in the database are inputs to qtag2latex.xml

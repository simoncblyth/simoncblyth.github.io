CHIBA TOMCAT¶

Integrating chiba into tomcat and pulling together the many chiba strands replaces the unmanaged $HEPREZ_FOLDER/xforms/chiba

TOMCAT_FOLDER : $TOMCAT_FOLDER
TOMCAT_HOME   : $TOMCAT_HOME
CHIBA_FOLDER  : $CHIBA_FOLDER
CHIBA_HOME    : $CHIBA_HOME

BUILDING¶

Prior to invoking chibatomcat-build must have run:

tomcat-build
chibacvs-build
chibatomcat-ant

invokes $(heprez-home)/chibatomcat/build.xml from $(chibatomcat-dir) NOT invoked by the -build

ancient build.xml default target of “chiba-mods” is dealing in non-existant relative paths “src/chiba-mods”

DOM jar shuffling is also there, but not done by default target

chibatomcat-build

Invokes the below:

chibatomcat-unwar
chibatomcat-customize
chibatomcat-deploy
chibatomcat-unwar

unpack the chiba-web distribution war chibacvs-distwar : $(chibacvs-distwar) twice into $(chibatomcat-dir)/{0,2}

Warning

CAUTION this will blow away form modications requiring a subsequent regeneration of the forms from the schema

chibatomcat-unwar-again
removes $(chibatomcat-dir)/2 and unwars into it again, bring chibatomcat back to pristine out-of-dist state
chibatomcat-customize
not implemented
chibatomcat-deploy

deploy “chiba.xml” context into $(tomcat-webapps)/ this hooks up chiba jars/classes/resources (eg xforms)

Example “chiba.xml”:

<Context path="/chiba" docBase="/data/heprez/install/chibatomcat/2" debug="0" privileged="true">
  <!-- Link to the user database we will get roles from -->
  <ResourceLink name="users" global="UserDatabase" type="org.apache.catalina.UserDatabase"/>
</Context>
chibatomcat-context-
emit tomcat context element to stdout this relates url path /chiba with webapp docbase
